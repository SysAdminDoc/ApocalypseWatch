# ApocalypseWatch Feature Backlog

| Feature | Description | Why it matters | Effort | API/server changes required | Risks / dependencies |
| --- | --- | --- | --- | --- | --- |
| Freshness confidence ribbon | Add a compact, always-visible score combining sample age, provider errors, archive validity, and last successful poll. | Viewers can quickly tell whether the dial is based on fresh telemetry or degraded data. | S | Client-only, preferable; uses existing `liveStatus`, archive, and fetch state. | Needs careful copy so it does not look like a second emergency score. |
| Data gap calendar | Show a 365-day miniature coverage grid for missing, delayed, or malformed half-hour samples. | Long-term readers can distinguish real calm periods from blind spots in the feed. | M | Client-only if archive timestamps remain enough to infer gaps. | RLE decode must stay cheap and avoid chart-bundle regressions. |
| Signal provenance drawer | Add a drawer that explains the current dial calculation step by step: current count, expected count, std dev, sigma, threshold, and timestamp. | Builds trust by making the emergency level auditable without reading source code. | S | Client-only, preferable; current payload already contains the needed composite signal fields. | Must avoid overclaiming causality from an intentionally simple model. |
| Cohort composition view | Break down the tracked fleet by model family, registration country, operator class, and passenger-capacity bands. | Users can understand what kind of aircraft population the signal represents before trusting anomalies. | M | Server/API change likely needed for richer FAA metadata unless already present in live records. | FAA registry normalization and operator inference can be messy. |
| Watchlist drift inspector | Show when the FAA-derived cohort was last imported and whether tail numbers have gone stale, duplicated, or lost registry metadata. | Prevents stale cohort data from silently weakening the signal over months. | M | Server/API change required to expose import timestamp and cohort health counts. | Requires pipeline metadata that may not exist yet. |
| Incident timeline | Persist level transitions, provider outages, stale-sample windows, and alert sends as a chronological event stream. | Turns the dashboard from a live gauge into an operational record people can review after an event. | L | Server/R2 change required for durable event storage; client can render the timeline. | Needs retention policy and duplicate suppression. |
| Alert-channel health card | Surface RSS export freshness, Telegram delivery status, and last alert attempt directly in the dashboard. | Readers can see whether external notification paths are working before a crisis. | M | Server/API change required; Telegram script currently does not write delivery state into `/api/dashboard`. | Credential and rate-limit failures must not leak secrets. |
| Public evidence packet | Generate a shareable current-state packet with dial value, data age, key counts, archive validation, and source links. | Analysts and journalists can cite the state of the dashboard without copying raw UI text by hand. | M | Client-only for JSON/text export; server change only if hosted permalink storage is needed. | Must avoid screenshots and ensure timestamps/timezones are explicit. |
| Sensitivity sandbox | Let users preview how the current and historical archive would score under alternate sigma thresholds without changing production behavior. | Makes the model’s assumptions transparent and helps calibrate false-positive concerns. | M | Client-only, preferable; uses existing archive and signal fields. | Could confuse users if sandbox state looks like the live emergency level. |
| Low-bandwidth mode | Add a query-param or preference that skips map/chart chunks and renders only gauge, status, and essential counts. | Keeps the dashboard usable during outages, on mobile data, or from constrained regions. | S | Client-only, preferable. | Must keep default experience unchanged and avoid fragmenting QA paths. |
| Wallboard mode | Provide a dense, non-interactive NOC display mode with larger freshness, level, and last-transition indicators. | Makes the dashboard useful on a shared operations screen without hover or keyboard interaction. | S | Client-only, preferable. | Needs burn-in-safe styling and careful reduced-motion behavior. |
| Corroboration layer | Compare ADS-B Exchange heatmap observations with one or more independent public aviation data sources when available. | Reduces the chance that a provider artifact looks like a real fleet movement anomaly. | L | Server/data-pipeline change required; not possible from the frozen client alone. | Provider access, licensing, latency, and schema mismatch are the main blockers. |
